## The concept

The **SOLID** principles are a set of five design principles intended to make
software designs more understandable, flexible, and maintainable.
Promoted by Robert C. Martin (Uncle Bob), they can form a core philosophy
for methodologies such as agile software development. These principles work hand-in-hand with [design patterns](/my-learning-curve/blog/20-design-patterns) to create robust, maintainable code.

> SOLID is a mnemonic for five design principles intended to make software
> designs more understandable, flexible, and maintainable.
>
> ~ Wikipedia

## The Principles

### S - Single Responsibility Principle (SRP)

A class (or function/component) should have only one reason to change.

#### OOP Example

```typescript
// ❌ Bad: Class handles two different responsibilities (business logic and logging/db)
class UserService {
  public createUser(user: User): void {
    // ... DB logic ...
    console.log('User created') // Logging responsibility mixed in
  }
}

// ✅ Good: Separation into distinct classes
class UserRepository {
  public save(user: User): void {
    /* ... */
  }
}
class Logger {
  public log(msg: string): void {
    /* ... */
  }
}
```

#### In FP Land

In Functional Programming (FP), pure functions are naturally aligned with SRP. Because an ideal pure function only takes inputs and returns an output without hidden side effects, it is forced to do exactly one focused thing.

```typescript
// ❌ Bad: Function combines logic and side effects
const processUser = (json: string) => {
  const user = JSON.parse(json)
  database.save(user) // Side effect inside
}

// ✅ Good: Pure function for parsing, separate function for side effects
const parseUser = (json: string): User => JSON.parse(json)
const saveUser = (user: User) => database.save(user)
```

### O - Open/Closed Principle (OCP)

Software entities should be open for extension, but closed for modification.

You should be able to add new functionality without changing existing code.

#### OOP Example

```typescript
// ❌ Bad: Modifying the base calculator for every new shape
class AreaCalculator {
  public calculate(shape: any): number {
    if (shape.type === 'circle') return Math.PI * shape.radius ** 2
    if (shape.type === 'square') return shape.side ** 2
    return 0
  }
}

// ✅ Good: Extending behavior by adding new classes that implement an interface
interface Shape {
  calculateArea(): number
}
class Circle implements Shape {
  constructor(public radius: number) {}
  calculateArea() {
    return Math.PI * this.radius ** 2
  }
}
class AreaCalculator {
  public calculate(shape: Shape): number {
    return shape.calculateArea()
  }
}
```

#### In FP Land

FP uses union types and pattern matching to handle variations in behavior. Instead of extending base classes, the compiler should enforce handling all possible variations of a union type cleanly. Adding new cases flips the "expression problem" structure, often making sure you handle new data across all matchers safely without modifying the inner logic of old data cases.

```typescript
// ✅ Good: Discriminated Unions with pattern matching and exhaustiveness checking
type Shape =
  | { type: 'circle'; radius: number }
  | { type: 'square'; side: number }

const calculateArea = (shape: Shape): number => {
  switch (shape.type) {
    case 'circle':
      return Math.PI * shape.radius ** 2
    case 'square':
      return shape.side ** 2
    default:
      // The compiler forces you to update this function safely if `Shape` changes.
      const _exhaustiveCheck: never = shape
      return _exhaustiveCheck
  }
}
```

### L - Liskov Substitution Principle (LSP)

Objects of a superclass shall be replaceable with objects of its subclasses
without breaking the application.

If you have a generic `Database` class, swapping it for `PostgresDatabase` or
`MongoDatabase` shouldn't crash your app if they implement the same interface.

#### OOP Example

```typescript
// ❌ Bad: Subclass throws an exception for a method defined in the parent
class Bird {
  fly(): string {
    return 'I am flying'
  }
}
class Penguin extends Bird {
  fly(): string {
    throw new Error('I cannot fly') // Violates LSP
  }
}

// ✅ Good: Hierarchy ensures correct structural substitutions
class Bird {}
class FlyingBird extends Bird {
  fly(): string {
    return 'I am flying'
  }
}
class Penguin extends Bird {}
```

#### In FP Land

In strict typed functional languages where there is no hidden mutation, no nulls, and errors are handled deliberately (e.g., using `Result` or `Option` types), LSP violations become structurally impossible. A function signature dictates exactly what can be substituted, so changing the underlying implementation logic guarantees the same input-to-output contract.

```typescript
// ❌ Bad: Passing a function that violates the contract by throwing unexpectedly
type CalculateDiscount = (price: number) => number

const standardDiscount: CalculateDiscount = price => price * 0.1
const specialDiscount: CalculateDiscount = price => {
  if (price < 0) throw new Error('Invalid price') // Hidden side-effect/failure!
  return price * 0.2
}

// ✅ Good: Encode failure modes explicitly in the type signature (Result/Union)
type CalculateDiscountSafe = (price: number) => number | Error
```

### I - Interface Segregation Principle (ISP)

Clients should not be forced to depend upon interfaces that they do not use.

Don't create massive interfaces. Break them down.

#### OOP Example

```typescript
// ❌ Bad: One massive interface forcing unused implementations
interface Machine {
  print(doc: Document): void
  fax(doc: Document): void
  scan(doc: Document): void
}

class BasicPrinter implements Machine {
  print(doc: Document) {
    /* ... */
  }
  fax(doc: Document) {
    throw new Error('Not supported')
  }
  scan(doc: Document) {
    throw new Error('Not supported')
  }
}

// ✅ Good: Segregated, specific interfaces
interface Printer {
  print(doc: Document): void
}
interface Scanner {
  scan(doc: Document): void
}

class BasicPrinter implements Printer {
  print(doc: Document) {
    /* ... */
  }
}
```

#### In FP Land

A function type is effectively a single-method interface—the smallest and most segregated interface possible. You literally cannot make a function type "fatter" unless you group them into bloated records or config objects. ISP is just how standalone functions behave by default.

```typescript
// ❌ Bad: Expecting a monolithic object when you only need a single property
interface UserContext {
  userId: string
  email: string
  preferences: Record<string, unknown>
  history: Record<string, unknown>[]
}

const sendWelcomeEmail = (context: UserContext) => {
  mailer.send(context.email, 'Welcome!')
}

// ✅ Good: Function signature dictates exactly (and only) what it needs
const sendWelcomeEmail = (email: UserContext['email']) => {
  mailer.send(email, 'Welcome!')
}
```

### D - Dependency Inversion Principle (DIP)

Depend upon abstractions, not concretions.

High-level modules should not depend on low-level modules. Both should depend on
abstractions.

#### OOP Example

```typescript
// ❌ Bad: High-level module depends on low-level concrete classes
class MySQLDatabase {
  save(data: string) {}
}
class UserService {
  private db = new MySQLDatabase() // Tigthly coupled
  saveUser(user: string) {
    this.db.save(user)
  }
}

// ✅ Good: Depend on abstractions via dependency injection
interface Database {
  save(data: string): void
}

class UserService {
  constructor(private db: Database) {} // Injection
  saveUser(user: string) {
    this.db.save(user)
  }
}
```

#### In FP Land

In a functional world, a function parameter _is_ the abstraction. When a function accepts another function as an argument, it depends only on its signature constraint, entirely decoupled from its concrete implementation. This makes Partial Application a natural and lightweight mechanism for Dependency Injection.

```typescript
// ❌ Bad: Hardcoded dependency on a concrete module/implementation
const saveUser = (user: User) => {
  mySqlInsert('users', user)
}

// ✅ Good: Inject logic via functions (Higher-Order Functions / Partial Application)
type SaveToDb = (table: string, data: unknown) => void

const makeSaveUser = (dbInsert: SaveToDb) => (user: User) => {
  dbInsert('users', user)
}

// Usage (Wiring up the dependencies at the composition root)
const saveUser = makeSaveUser(mySqlInsert)
```
